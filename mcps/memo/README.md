# memo

タイトルと概要を持つメモを管理する、シンプルな MCP サーバー。SQLite にメモを保存し、作成・取得・一覧・更新・削除を MCP ツールとして提供する。

## 特徴

- **2 種類の検索**: タイトルの部分一致検索（カンマ区切りで複数キーワードの OR 検索）と、概要の意味的な近さで探すセマンティック検索（OpenAI 埋め込み）。
- **カテゴリ（ユーザーごとに管理）**: カテゴリは各ユーザーが所有する独立した存在で、メモは**自分が登録済みのカテゴリにだけ**紐づけられる（未登録カテゴリを指定するとエラー）。新規ユーザーは既定の `OTHERS` だけを持ち、必要なカテゴリを追加していく。カテゴリ名は大文字に正規化される。一覧・検索時にカテゴリを指定すると同一カテゴリのメモだけに絞り込める。カテゴリの**追加・閲覧は MCP ツールから**、**追加・改名・削除（CRUD）は Web 画面から**できる。改名するとそのカテゴリに紐づくメモも自動で追従し、削除すると紐づくメモは `OTHERS` に移る（`OTHERS` は改名・削除不可）。
- **ユーザーごとに完全分離**: メモには作成した接続ユーザーが所有者として記録され（不変の内部 ID で紐づく）、他ユーザーは読み取りも含め一切操作できない。管理者も例外ではない。
- **ユーザー登録制**: 接続ユーザーは `users` 台帳に登録されている必要があり、未登録の接続はすべてのツール呼び出しが拒否される。
- **ユーザー ID と管理者権限フラグ**: 各ユーザーは不変の内部 ID（`id`）を持ち、メモ・カテゴリはこの ID に紐づく（ユーザー名 `name` はログイン用のハンドルで、将来名前を変えてもメモを付け替える必要がない設計）。**管理者かどうかは名前ではなく独立した `is_admin` フラグで管理する**——名前が `admin` でなくてもフラグを立てれば管理者にできる。`is_admin` の編集は後述の管理 Web UI からのみ行える。**最後の 1 人の管理者は削除も降格もできない**（管理者が居なくなるのを防ぐ）。
- **admin の役割**: 管理者は**ユーザーの登録・編集・削除（ユーザー管理）ができるだけ**で、他人のメモやカテゴリは一切操作できない。既定の管理者ユーザー `admin`（`is_admin=1`）は DB 初期化時に自動登録される。
- **admin ツールの可視性**: ユーザー管理ツール（`create_user` など 5 つ）は**サーバーレベルで既定無効**で、一覧に出ず呼び出しても拒否される。`switch_user` で**管理者（`is_admin`）のユーザー**に切り替えた接続でだけ、その接続限定で有効化される（FastMCP の公式コンポーネント可視性機能。クライアントには `list_changed` 通知が自動で飛ぶ）。この自動有効化は環境変数で無効化もできる（後述）。
- **接続を張り替えずにユーザー切り替え**: `switch_user` ツールで、再接続やサーバー再起動なしに現在のユーザーを切り替えられる（個人ローカル運用向け）。
- **監査ログ**: ツール呼び出しを 1 箇所で記録する。`--debug` で `Mcp-Session-Id` を含むセッション全体像を出力できる。

提供するツールの一覧・使い方は [TOOLS.md](./TOOLS.md)、システムプロンプトへの組み込み例は [USECASE.md](./USECASE.md)、データモデルや実装の詳細は [CLAUDE.md](./CLAUDE.md) を参照。

## 実行

```bash
# stdio トランスポート（Claude Desktop / VS Code はこの方式で接続する）
uv run memo --user alice

# HTTP トランスポート。接続側は /mcp?user=NAME&client_id=ID を指定する
TRANSPORT=http PORT=8080 uv run memo

# セッション全体像（Mcp-Session-Id など）を DEBUG ログで見たいとき
uv run memo --user alice --debug          # 環境変数 MEMO_LOG_DEBUG=1 でも可
```

DB ファイルの場所は環境変数 `MEMO_DB_PATH` で上書きできる（デフォルト: `src/memo/memo.db`）。HTTP のバインドアドレスは `HOST`（デフォルト `127.0.0.1` = ローカル束縛）、ポートは `PORT`（デフォルト `8080`）。

### データベースのマイグレーション

スキーマの変更は**バージョン管理付きマイグレーション**で管理する（`src/memo/migrations/`）。`schema_version` テーブルで適用済みバージョンを記録し、未適用のものだけを順に適用する。サーバー起動時に毎回同じ処理（スキーマ作成 → マイグレーション → 初期シード）が冪等に走るので、通常は意識する必要はない。

既存 DB を手動で最新スキーマへ移行・確認したいときは次を使う（冪等。移行前後の `schema_version` を表示する）。

```bash
uv run memo-migrate                 # MEMO_DB_PATH / 既定の memo.db を対象に移行
uv run memo-migrate /path/to/memo.db
```

## ユーザーの識別と登録

すべてのメモは「接続ユーザー」が所有する。ユーザーはトランスポートごとに異なる方法で渡す。

| トランスポート | 指定方法 | 例 |
|---------------|---------|----|
| stdio | 起動引数 `--user`（または環境変数 `MEMO_USER`）。プロセス全体でユーザーは 1 人に固定される | `uv run memo --user alice` |
| HTTP | MCP エンドポイントのクエリパラメータ `?user=`。任意で `?client_id=` を併用 | `http://host:8080/mcp?user=alice&client_id=desktop-1` |

接続が許可されるのは `users` 台帳に登録済みのユーザーのみ。ユーザーを識別できない（`--user` / `?user=` の指定が無い）、または識別できても未登録の接続は、すべてのツール呼び出しがエラーで拒否される。新しいユーザーは `admin` で接続して `create_user` ツールで登録する。あるいは、後述の管理 UI（`memo-admin`）からブラウザで登録・編集・削除できる。

### ユーザー管理 UI（`memo-admin`）

DB を直接いじったり Claude 経由で `create_user` を呼んだりせずに、ブラウザから `users` 台帳をサッと管理するためのローカル Web サーバー。MCP サーバー本体とは**別プロセス**で動き、同じ `memo.db` を読み書きする（SQLite は WAL モードなので別プロセスからの同時アクセスでも安全）。Claude Desktop を起動していなくても単独で使える。

```bash
# 管理画面を起動して http://127.0.0.1:8090 を開く
uv run memo-admin
```

画面ではユーザーの一覧表示・追加・表示名/備考の編集・削除に加え、**管理者権限（`is_admin`）の付与・剥奪**ができる（`is_admin` を編集できるのはこの Web UI だけ）。一覧には「管理者」列のチェックボックスがあり、保存すると反映される。ユーザー追加フォームにも「管理者」チェックボックスがある。ユーザー名 `name` は不変なので編集対象は `display_name` / `note` / `is_admin`。**最後の 1 人の管理者は削除も降格もできない**（API 側でも拒否される）。バインド先は環境変数 `MEMO_ADMIN_HOST`（既定 `127.0.0.1`）、ポートは `MEMO_ADMIN_PORT`（既定 `8090`）で変更できる。DB ファイルは MCP サーバーと同じく `MEMO_DB_PATH` に従う。

各ユーザー行の「メモ」ボタンを押すと、そのユーザーのメモ一覧を新しい順（更新日時の降順）に表示する。メモは大量になり得るため**ページング**に対応し（1 ページ 20 件、「前へ / 次へ」で移動）、現在ページと総件数を表示する。このメモ画面では、そのユーザーのメモを**作成・編集・削除**できる。

- **作成**: 一覧の「新規メモ」ボタンを押すと編集画面に切り替わり、タイトル（必須）・カテゴリ（そのユーザーの登録済みカテゴリから選択。既定は `OTHERS`）・概要（任意）を入力して「保存」。作成したメモはそのユーザーの所有になる。
- **編集**: 各行の「編集」ボタンを押すと編集画面に切り替わり、その内容を読み込んだ状態でタイトル／カテゴリ／概要を直して「保存」する（タイトルは空にできない）。「キャンセル / 戻る」で保存せず一覧に戻れる。
- **カテゴリ管理**: メモ画面の「カテゴリ管理」ボタンから、そのユーザーのカテゴリを**追加・改名・削除**できる。改名するとそのカテゴリに紐づくメモも自動で追従し、削除すると紐づくメモは `OTHERS` に移る。既定カテゴリ `OTHERS` は改名・削除できない。
- **カテゴリ絞り込み**: メモ一覧の上部のカテゴリ選択（プルダウン）で同一カテゴリのメモだけを表示できる。「(全カテゴリ)」を選ぶと絞り込み解除。各行にはカテゴリがバッジで表示される。
- **削除**: 各行の「削除」で確認のうえ消す（画面遷移はしない）。

一覧の行は表示専用で、編集・作成は専用の編集画面で行う。操作は常に表示中のユーザーのメモ／カテゴリに限られ、他ユーザーのデータには一切触れられない（完全分離）。

> **セキュリティ**: この管理画面は `switch_user` と同様に**無認証**で、`admin` を含む全ユーザーを操作できる。既定の `127.0.0.1` バインドはローカル個人運用が前提。`MEMO_ADMIN_HOST=0.0.0.0` などで外部公開する場合は、必ず前段に認証を置くこと。

### ユーザーの切り替え（`switch_user`）

再接続やサーバー再起動をせずに現在のユーザーを切り替えるには `switch_user` ツールを使う。切り替え先は登録済みユーザーである必要がある（`admin` への切り替えも可能）。

- **stdio**: `switch_user("alice")` を呼ぶと、以後この接続のメモ操作が alice のものになる（プロセス内で実行時に切り替わる）。
- **HTTP**: 接続時のクエリに `?client_id=` を付けておくこと。サーバーは「安定した接続識別子 `client_id`」と「現在のユーザー」を対応づけて保持し、`switch_user` でこの対応を書き換える。これにより `?user=` の値に関わらず、また**接続を張り替えても**（`Mcp-Session-Id` は再接続で変わるが `client_id` は不変）、切り替えたユーザーが維持される。`client_id` が無い HTTP 接続では切り替え状態を保持できないためエラーになる。

> 対応づけはメモリ上に保持されるため、サーバーを再起動すると初期値（`?user=`）に戻る。

### admin ツールの可視性（サーバーレベルの既定 ＋ セッション単位の有効化）

ユーザー管理ツール（`create_user` / `get_user` / `list_users` / `update_user` / `delete_user`）は破壊的になり得るため、FastMCP の公式「コンポーネント可視性」機能で 2 段構えに制御している。

- **サーバーレベル（既定無効）**: 起動時にこれらを無効化する。よって管理者で起動した接続を含め、**どの接続も最初は一覧に現れず、名指しで呼び出しても `Unknown tool` で拒否される**（存在も漏らさない）。
- **セッション単位（管理者切替で有効化）**: `switch_user` で現在のユーザーが**管理者（`is_admin`）**に切り替わると、**その接続だけ**で管理ツールが有効化される（再接続不要。クライアントには `list_changed` 通知が自動送信されるので、ツール一覧が更新されて有効化が反映される）。管理者でないユーザーに切り替え直すと再び無効化される。判定は名前ではなく `is_admin` フラグで行う。

トリガは `switch_user` なので、管理者で起動した接続でも一度 `switch_user` で管理者ユーザーへ切り替えると管理ツールが有効になる（自分自身への切り替えで可視性が再適用される）。`switch_user` 自体は admin 専用ではないため常に見える（これが管理者へ到達する入口）。

この「管理者に切り替えると管理ツールが生える」自動有効化は、誰でも無認証で `switch_user` できるローカル運用では危険になり得る。これを丸ごと止めたい場合は、環境変数 `MEMO_ADMIN_TOOLS_AUTO_ENABLE` を `false`（`0` / `no` / `off` でも可）にする。無効化すると、管理者に切り替えても管理ツールは有効化されない。

```dotenv
# mcps/memo/.env（環境変数が優先され、無ければ .env にフォールバック）
# 管理者に切り替えても管理ツールを有効化しない（既定は有効＝管理者切替で公開する）
MEMO_ADMIN_TOOLS_AUTO_ENABLE=false
```

> 管理ツールが無効化されているときでも、`memo-admin` の Web UI からはユーザーの登録・編集・削除ができる。

## セマンティック検索のセットアップ

セマンティック検索（`semantic_search_memos`）を使うには OpenAI の API キーが要る。`mcps/memo/.env` に書いておけば起動時に自動で読み込まれる（既に設定済みの環境変数が優先され、.env は上書きしない）。

```dotenv
# mcps/memo/.env
OPENAI_API_KEY=sk-...
# MEMO_EMBEDDING_MODEL=text-embedding-3-small   # 任意（デフォルトは多言語対応モデル）
```

`.env` は `.gitignore` 済みなので API キーはコミットされない。キーが未設定でも他のツールは動作し、`semantic_search_memos` だけがエラーを返す。

## ログ・デバッグ

ツール呼び出しは監査ログとして 1 行ずつ記録される（出力先は **stderr**。stdio では stdout が MCP プロトコル本体に使われるため）。

- **既定（INFO）**: ツール呼び出しごとに `tool=… user=… client_id=… session=…` の 1 行。`session` は `Mcp-Session-Id`。
- **`--debug`（または `MEMO_LOG_DEBUG=1`）**: `initialize` を含む全メソッドについて、`request_id` / `client_id` / 解決前の `?user=`（`raw_user`） / 解決後のユーザー / `session` を出力する。`Mcp-Session-Id` が接続を通じて安定し、再接続で張り替わる様子を観察できる。DEBUG 化されるのは `memo.*` のログだけで、ライブラリの DEBUG ノイズは混ざらない。

## 注意点

- セマンティック検索の埋め込みは検索時に遅延計算してキャッシュされる。初回に未キャッシュのメモが多いと、その数だけ OpenAI API を呼ぶ。
- **親テーブルの変更には DB の仕組みで追従する**（手動更新はしない）。外部キーを常に有効化しており、ユーザーを削除するとそのカテゴリ・メモ・埋め込みキャッシュが、メモを削除するとその埋め込みキャッシュが、それぞれ**自動でまとめて削除される**（孤立データを残さない）。カテゴリの改名・削除によるメモ側の追従（改名は付け替え、削除は `OTHERS` へ戻す）は DB のトリガーが行う。
- 既存 DB の移行は**バージョン管理付きマイグレーション**で行う（後述）。外部キーが無かった時代の DB は初回起動時（または `memo-migrate`）に、まず外部キー付き（名前ベース）へ、続いて**ユーザー ID ベース**（`users` に不変の `id` と `is_admin` を追加し、メモ・カテゴリの所有者参照を名前から ID に付け替え。既存の `admin` 行には `is_admin=1` が付く）へ自動で作り替えられる。その際に所有者不明のメモや実体を失った埋め込みなどの孤立データは取り除かれる。`categories` テーブルへの各ユーザーの `OTHERS` と既存メモのカテゴリの登録は従来どおり初期化時に自動で行われる。
- **HTTP の公開範囲**: 既定の `HOST=127.0.0.1` はローカル束縛で、個人ローカル運用を前提とする。`switch_user` は認証なしで管理者を含む任意ユーザーに切り替えられるため、`HOST=0.0.0.0` で外部公開するとネットワーク上の誰でも管理者化できてしまう。共有・公開する場合は前段に認証（例: [dynamic_prompt](../dynamic_prompt/) の Google OAuth 相当）を必ず置くこと。
